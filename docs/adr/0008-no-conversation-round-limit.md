# ADR-0008：对话轮次取消上限，用户主导终止

> 日期：2026-06-27 | 状态：已采纳 | 决策者：产品裁决（/grill-with-docs 领域建模）

---

## 背景

PRD v1.0 和 CLAUDE.md 均规定"8 轮硬熔断"——对话 Agent 在 `conversation_round >= 8` 时强制终止对话并触发报告生成。该规则来自早期产品假设：诊断信息可以在有限轮次内收集完毕。

但在实际开发中暴露出三个矛盾：

1. **信息质量 vs 轮数**：用户可能在 5 轮时已经提供足够信息，也可能在 8 轮时仍在补充关键细节。一刀切轮数会产生过早终止或被迫打断的问题。
2. **代码从未实现**：`should_stop_conversation()` 始终返回 `False`；`max_rounds=8` 字段存在但从未被读取。
3. **用户体验**：自然对话的核心价值是"像跟真人顾问聊天"，对方不会在刚好聊了 8 句时说"时间到了，强制结束"。

---

## 决策

**对话轮次不再有上限。** 终止由用户主导，系统只做建议和门槛检查。

具体规则：

| 终止触发 | 优先级 | 行为 |
|---------|:------:|------|
| 用户点击"生成报告" | **最高** | 系统检查信息门槛。达标 → 正常评分 + 报告；不达标 → 返回缺失项列表，不生成 0 分模板 |
| 信息完整度提示 | 建议 | 当 `answers ≥ 8` 且关键字段就绪，前端提示"可以生成报告"，但不强制终止 |
| AI 连续失败 | 技术兜底 | `ai_failure_count ≥ 2` 时进入 `fallback_questionnaire`。这是技术故障路径，不是正常业务终止 |

明确废弃的概念：

- `max_rounds` 字段 → 不再参与终止逻辑，改为仅统计展示或直接移除
- `should_stop_conversation()` → 函数名已误导，保持返回 `False`，后续移除或改名
- PRD 中"8 轮硬熔断" → 已过期
- CLAUDE.md 中"8 轮硬熔断" → 已过期

---

## 替代方案

**方案 B：保留 8 轮硬上限，但允许"槽位满后提前终止"**
- 优点：降低无限对话的 token 成本风险
- 否决原因：仍然存在"槽位不满但到 8 轮"的死角；且代码实现复杂度等同新方案

**方案 C：AI 判断是否终止（让 LLM 决定"信息够了"）**
- 优点：最接近自然对话体验
- 否决原因：DeepSeek 对"何时终止"的判断不稳定，会引入不可预测的用户体验（有时 3 轮就停，有时 15 轮不停）

---

## 后果

**正面**：
- 用户体验统一——对话由用户自己掌控
- 代码逻辑简化——移除了一条从未实现的终止路径
- 模板报告的触发场景收窄为"AI 技术故障"，不再用于"用户信息不够"

**负面**：
- 无限对话可能增加 token 成本（用户一直聊但不生成报告）
- 前端需要实现"信息完整度提示"的新 UI 状态

**需要的后续变更**：
1. 更新 PRD（`docs/superpowers/specs/2026-06-24-chuhai-agent-prd.md`）：删除 8 轮熔断
2. 更新 CLAUDE.md：删除 8 轮熔断描述
3. 前端实现"信息足够，可以生成报告"提示
4. 后端 `finish` 端点实现信息门槛检查 + 缺失项列表返回
5. 移除或标记废弃 `max_rounds` 字段
