# Vibecoding 项目构建顺序手册

## 1. 文档目的

这份手册不是某一个功能的设计文档，而是一套可跨项目复用的 **vibecoding 项目构建顺序**。目标是解决三个常见问题：

1. 先写什么，后写什么，避免一上来就让 AI “从零到上线一把梭”
2. 哪些文档必须先有，才能让 AI 在整个项目周期里持续稳定地产出
3. 如何把项目拆成可验证、可回滚、可测试的小步迭代，而不是做出一堆看起来很对、实际上跑不通的代码

本手册以当前 LakeQuery 这类 **AI Agent / NL2SQL / 后端+前端+外部依赖** 项目为参照，总结出一套通用顺序。以后你做别的 vibecoding 项目，也可以直接套用。

---

## 2. 一句话原则

**先收敛问题，再约束 AI；先写文档，再起骨架；先打通最小垂直切片，再扩展能力；先 mock，再接真实依赖；先测试，再实现；先安全，再上线。**

如果只记得一条，就是这条。

---

## 3. 总体构建顺序总览

| 阶段 | 名称 | 核心问题 | 主要产物 | 完成标志 |
| --- | --- | --- | --- | --- |
| 0 | 立项收敛 | 我到底要做什么，不做什么 | PRD 初稿 | 目标用户、场景、边界清楚 |
| 1 | AI 约束落地 | 以后 AI 按什么规范写代码 | AGENTS.md | AI 有稳定工作边界 |
| 2 | 技术设计 | 系统怎么拆，模块怎么连 | TECH_DESIGN.md | 架构、模型、API、测试策略完整 |
| 3 | 文档底座 | 让项目可运行、可协作、可复现 | README.md、.env.example、依赖文件 | 新人能按文档启动 |
| 4 | 测试与评估基线 | 怎么验证 AI 没胡写 | tests/、fixtures、评估样本 | 至少有一条红灯测试链路 |
| 5 | 工程骨架 | 项目目录、配置、入口怎么搭 | 代码骨架、空实现、启动脚本 | 后端/前端/测试能启动 |
| 6 | 最小垂直切片 | 第一个真正可跑通的用户价值 | 单条 happy path 端到端 | 用户能完成一次完整流程 |
| 7 | 核心能力扩展 | 把主功能逐步补全 | 多模块实现 + 测试 | PRD 的 P0 功能闭环 |
| 8 | 真实依赖接入 | 从 mock/demo 切到真实环境 | 外部系统接入文档与实现 | 在真实依赖上成功跑通 |
| 9 | 稳定性与安全 | 代码能不能扛住错误输入 | 安全规则、重试、限流、日志 | 异常路径有防护 |
| 10 | 交互与运维 | 用户怎么用，运维怎么管 | UI、部署、监控、Runbook | 可以交付别人试用 |
| 11 | 上线前收口 | 还有什么风险未消除 | Release checklist、审计报告 | 满足发版门槛 |
| 12 | 复盘沉淀 | 下次能不能更快更稳 | DEVELOPMENT_JOURNEY.md | 问题和经验被沉淀 |

这 13 个阶段不是严格瀑布模型，而是主顺序。可以回退，但不能跳过前置约束直接往后冲。

## 4. 文档清单总表

下面是 vibecoding 项目建议维护的文档清单。不是每个项目都要一次性建全，但顺序要对。

| 文件 | 是否必需 | 创建时机 | 作用 |
| --- | --- | --- | --- |
| PRD.md | 必需 | 阶段 0 | 定义要做什么、用户是谁、成功标准是什么 |
| AGENTS.md | 必需 | 阶段 1 | 约束 AI 的代码风格、架构边界、测试规则 |
| TECH_DESIGN.md | 必需 | 阶段 2 | 定义技术栈、模块边界、数据结构、API、测试方案 |
| README.md | 必需 | 阶段 3 | 项目介绍、启动方式、目录结构、常用命令 |
| .env.example | 必需 | 阶段 3 | 所有环境变量的模板与说明 |
| requirements.txt / pyproject.toml | 必需 | 阶段 3 | 后端依赖声明 |
| package.json | 有前端时必需 | 阶段 3 | 前端依赖声明 |
| docker-compose.yml | 强烈建议 | 阶段 3 或 10 | 拉起依赖服务，保证可复现 |
| Dockerfile | 强烈建议 | 阶段 3 或 10 | 保证部署环境一致 |
| pytest.ini | 强烈建议 | 阶段 4 | 固化测试约定 |
| tests/fixtures/* | 必需 | 阶段 4 | 给 AI 稳定、可重复的测试数据 |
| API_SPEC.md | 视复杂度决定 | 阶段 2 或 3 | 复杂 API 项目建议单独维护 |
| TEST_PLAN.md | 视复杂度决定 | 阶段 4 | 对测试范围、优先级、评估口径做单独说明 |
| SECURITY.md | 强烈建议 | 阶段 9 | 收口威胁模型、安全假设、防护边界 |
| OPERATIONS_RUNBOOK.md | 强烈建议 | 阶段 10 | 运维手册、排障流程、告警说明 |
| RELEASE_CHECKLIST.md | 强烈建议 | 阶段 11 | 发版前逐项检查清单 |
| DEVELOPMENT_JOURNEY.md | 建议 | 阶段 12 | 复盘项目演化过程和关键坑点 |
| CHANGELOG.md | 公开项目建议 | 阶段 11 起 | 对外记录版本变化 |

### 4.1 文档最小集合

如果你做的是小型 vibecoding 项目，最少要有：

1. PRD.md
2. AGENTS.md
3. TECH_DESIGN.md
4. README.md
5. .env.example
6. tests/ + fixtures

这六样没有，AI 基本一定会越写越飘。

### 4.2 中型 Agent 项目的推荐集合

对于当前这种 AI Agent / NL2SQL / 多依赖项目，推荐至少有：

1. PRD.md
2. AGENTS.md
3. TECH_DESIGN.md
4. README.md
5. .env.example
6. requirements.txt
7. docker-compose.yml
8. Dockerfile
9. pytest.ini
10. tests/fixtures/
11. RELEASE_CHECKLIST.md
12. DEVELOPMENT_JOURNEY.md

## 5. 每个阶段该怎么做

## 5.0 阶段 0：先收敛问题，不要先写代码

### 目标

把项目问题空间收敛到 AI 可以准确理解的程度。

### 要做什么

1. 明确用户是谁
2. 明确核心场景是什么
3. 明确第一期只做什么，不做什么
4. 明确成功指标是什么
5. 明确最关键的技术风险是什么

### 产物

1. PRD.md

### PRD 最少应该包含

1. 产品概述
2. 用户画像
3. 背景与痛点
4. 功能分期
5. 非功能需求
6. 技术选型草案
7. 成功指标
8. 里程碑计划
9. 风险与应对

### 这个阶段不要做什么

1. 不要开始写类、写接口、写 SQL、写前端页面
2. 不要在没有边界的情况下让 AI “直接搭项目”
3. 不要过早讨论 UI 细节、监控平台、K8s 等后期问题

### 给 AI 的标准提问模板

```text
先不要写代码。请根据我的项目想法，和我一起把 PRD 收敛完整，至少覆盖目标用户、核心场景、第一期功能、非功能需求、技术风险和成功指标。
如果你发现存在关键前提不清楚，先提问再写文档。
```

### 完成标志

你能用 3 句话说清楚：

1. 项目给谁用
2. 项目解决什么问题
3. 第一版成功长什么样

---

## 5.1 阶段 1：用 AGENTS.md 把 AI 关进笼子里

### 目标

让 AI 后续所有代码生成都遵守同一套工程规范，而不是每次对话重新解释。

### 产物

1. AGENTS.md

### AGENTS.md 最少应该约束的内容

1. 语言版本和代码风格
2. 项目目录结构
3. 层级依赖规则
4. TDD 开发流程
5. 错误处理规范
6. Prompt 管理规范
7. Git 规范
8. 启动与测试命令

### 对 Agent 项目特别重要的规则

1. 节点函数签名必须统一
2. 状态模型必须集中定义
3. LLM 输出必须结构化
4. 所有核心能力必须先测后写
5. 外部依赖必须支持 mock

### 给 AI 的标准提问模板

```text
请根据 PRD，为这个项目创建 AGENTS.md。
重点约束：目录结构、层级依赖、测试先行、LLM 输出结构化、安全底线、Prompt 管理方式。
这个文件以后要作为整个项目 vibecoding 的总规范。
```

### 完成标志

以后你让 AI 写任何代码时，只要说“按 AGENTS.md 来”，它就知道不能乱来。

---

## 5.2 阶段 2：技术设计先于工程骨架

### 目标

在写代码前，把系统拆分清楚：模块、数据结构、API、状态流转、测试策略、依赖接入方式。

### 产物

1. TECH_DESIGN.md
2. 可选：API_SPEC.md、SECURITY.md、TEST_PLAN.md

### TECH_DESIGN.md 必须覆盖的内容

1. 技术栈
2. 项目目录结构
3. 核心数据结构
4. 状态机或主流程设计
5. API 设计
6. 配置管理
7. 安全设计
8. 测试规范
9. 部署方式

### Agent / AI 项目额外必须覆盖

1. LLM 调用入口统一在哪里
2. Prompt 放在哪里，MVP 怎么管理
3. 重试机制如何控制
4. 中间状态如何传递
5. Checkpointer 或 Memory 后续如何接入
6. Mock 数据和真实依赖的切换路径

### 给 AI 的标准提问模板

```text
请基于 PRD 和 AGENTS.md，写 TECH_DESIGN.md。
必须覆盖技术栈、数据结构、API 调用、主流程状态机、测试策略和部署方式。
文档要足够详细，保证后面可以按它一模块一模块实现。
```

### 完成标志

如果把代码全删掉，只保留 TECH_DESIGN.md，另一个工程师也能按文档重建系统。

---

## 5.3 阶段 3：补齐项目的可运行底座

### 目标

让项目具备最基本的“可安装、可启动、可配置、可协作”能力。

### 产物

1. README.md
2. .env.example
3. requirements.txt / pyproject.toml
4. package.json（如果有前端）
5. Dockerfile
6. docker-compose.yml
7. pytest.ini

### README.md 至少应该有

1. 项目简介
2. 核心能力
3. 项目结构
4. 快速开始
5. 测试命令
6. 部署方式
7. 常见问题或已知限制

### .env.example 至少应该有

1. 所有必须环境变量
2. 每个变量的注释说明
3. 默认值或示例值
4. 不同环境是否有差异

### 这个阶段最容易犯的错

1. 代码已经写了一堆，别人却启动不起来
2. 真实密钥写进代码或 README
3. Docker 能跑，本地跑不起来；或者反过来

### 完成标志

一个新同事拉下代码后，只看 README 和 .env.example 就能把项目启动起来。

---

## 5.4 阶段 4：先建测试地基，再让 AI 写实现

### 目标

把“AI 写的代码到底对不对”变成一个自动化可验证问题。

### 产物

1. tests/unit/
2. tests/integration/
3. tests/fixtures/
4. conftest.py
5. 可选：TEST_PLAN.md、EVAL_DATASET.md

### 推荐顺序

1. 先为核心数据结构写模型测试
2. 再为纯函数模块写单元测试
3. 再为流程控制写集成测试
4. 最后才接真实 API、真实向量库、真实数据库

### 对 AI 项目尤其重要的测试对象

1. 结构化输出解析
2. 状态机路由
3. 错误重试
4. 安全拦截
5. 配置读取
6. 外部依赖失败时的降级行为

### 最少要有的 fixture

1. 一组典型输入
2. 一组边界输入
3. 一组错误输入
4. 一组模拟外部依赖响应

### 给 AI 的标准提问模板

```text
先不要写实现。请按照 AGENTS.md 和 TECH_DESIGN.md，为 [某个模块] 先写 Pytest 测试。
至少覆盖 happy path、边界情况和一个异常 case。
测试写完后再实现最小代码让它通过。
```

### 完成标志

项目里已经出现第一批红灯测试，并且每个后续模块都有明确的测试入口。

---

## 5.5 阶段 5：搭工程骨架，但只搭空骨架

### 目标

建立项目目录、入口文件、配置对象、抽象接口和占位实现。

### 产物

1. 项目目录树
2. 配置模块
3. 主入口
4. 抽象基类 / 协议接口
5. 空节点、空路由、空服务实现

### 这里的原则

1. 只搭结构，不填大逻辑
2. 先让 import 和启动路径成立
3. 先让测试框架、配置框架、路由注册跑通

### 这个阶段不应该做的事

1. 不要在所有文件里塞满大段业务逻辑
2. 不要一开始就把真实第三方 SDK 深度耦合进去
3. 不要让 API、Agent、数据库逻辑互相直接调用

### 完成标志

1. 项目能启动
2. 测试命令能执行
3. 目录结构和文档一致

---

## 5.6 阶段 6：先打穿一条最小垂直切片

### 目标

用最小功能路径证明你的架构是真的可行，不是纸上谈兵。

### 什么叫最小垂直切片

不是“实现一个模块”，而是“让一个真实用户请求从入口走到出口”。

### 对 LakeQuery 这种项目，最小垂直切片应该是

1. 输入自然语言问题
2. 解析出简单意图
3. 命中本地 mock schema
4. 生成一条简单 SELECT SQL
5. 通过 AST 校验
6. 通过 DuckDB dry-run
7. 返回结果给 API

### 为什么这一步最重要

因为只有它能同时验证：

1. 架构是不是通的
2. 数据结构是不是够用
3. 模块边界是不是合理
4. 文档是不是可执行

### 完成标志

用户可以完成第一次端到端 happy path，即使功能很简陋也没关系。

---

## 5.7 阶段 7：围绕垂直切片扩展核心功能

### 目标

按 PRD 的优先级，把 P0 能力一项一项补齐。

### 推荐顺序

1. 先补“主链路中的缺口”
2. 再补“提高成功率的能力”
3. 最后补“提升体验的能力”

### 对 Agent / NL2SQL 项目的典型顺序

1. Schema 检索
2. SQL 生成
3. AST 校验
4. 沙盒校验
5. 自动纠错
6. 方言转译
7. 元数据问答
8. 黄金查询复用

### 每新增一个能力时的固定动作

1. 先补测试
2. 再补实现
3. 再补 README 或 TECH_DESIGN 中的对应说明
4. 再跑局部验证

### 完成标志

PRD 中所有 P0 功能都有对应模块、测试和文档落点。

---

## 5.8 阶段 8：按“假依赖 → 真依赖”的顺序接入外部系统

### 目标

避免一上来就被真实环境拖死。

### 通用接入顺序

1. 固定 fixture / 本地 JSON
2. mock client
3. 本地可运行依赖（Docker、本地模式）
4. 测试环境依赖
5. 生产级真实依赖

### 对 LakeQuery 这种项目的推荐顺序

1. 本地 JSON 表结构
2. mock LLM 返回结构化输出
3. 本地 Qdrant / 内嵌模式
4. 本地 DuckDB 沙盒
5. 真实 LLM API
6. 真实 HMS / Iceberg Catalog
7. 真实查询引擎执行

### 这样做的原因

1. 先验证代码结构，再验证网络和权限
2. 先暴露程序逻辑问题，再暴露环境问题
3. 先让系统在可控数据上稳定，再承受真实数据复杂度

### 完成标志

真实依赖接入的问题，和业务代码问题可以被明确区分开。

---

## 5.9 阶段 9：安全、稳定性、可观测性要在主功能跑通后立刻补上

### 目标

把“能跑”升级为“可控地跑”。

### 必须补的东西

1. 输入校验
2. 错误分类
3. 超时控制
4. 重试上限
5. 速率限制
6. 审计日志
7. 监控埋点
8. 安全规则文档

### 对 Agent 项目的关键稳定性点

1. LLM 返回空值怎么办
2. 结构化输出校验失败怎么办
3. 向量检索无结果怎么办
4. 外部数据库不可达怎么办
5. 重试超过阈值怎么办
6. 多轮状态丢失怎么办

### 推荐文档

1. SECURITY.md
2. OPERATIONS_RUNBOOK.md
3. 审计或稳定性报告

### 完成标志

错误不再只是“报栈”，而是能被分级、记录、回退、恢复。

---

## 5.10 阶段 10：再做 UI、部署、运维交付

### 目标

把“开发者能跑”升级成“用户和运维都能用”。

### 应包含的内容

1. 前端交互页面或控制台
2. 部署脚本与容器化
3. 健康检查接口
4. 日志与监控接入
5. 环境变量说明
6. 常见故障排查手册

### 推荐产物

1. Dockerfile
2. docker-compose.yml
3. OPERATIONS_RUNBOOK.md
4. 完整 README 启动说明

### 完成标志

不是只有你自己能用，而是别人也能根据文档启动、配置、测试和排障。

---

## 5.11 阶段 11：上线前收口，而不是边上线边补文档

### 目标

把所有未闭环风险显式列出来，并用清单控制发版质量。

### 必须检查

1. 测试是否全绿
2. 是否还有 mock 未替换
3. 是否仍有硬编码配置
4. 是否仍有未处理异常
5. 是否仍有安全风险未评估
6. 是否仍有 README 与真实实现不一致
7. 是否仍有前后端接口不一致
8. 是否已有最小回滚方案

### 推荐产物

1. RELEASE_CHECKLIST.md
2. 稳定性审查报告
3. 全局代码审计报告

### 完成标志

发版前不是“感觉差不多”，而是每一项都能对照文档打勾。

---

## 5.12 阶段 12：最后做复盘沉淀

### 目标

把这次 vibecoding 的坑、决策和演进路线记下来，给下一次项目复用。

### 推荐产物

1. DEVELOPMENT_JOURNEY.md
2. CHANGELOG.md
3. 关键设计决策记录

### DEVELOPMENT_JOURNEY.md 建议记录什么

1. 每个里程碑做了什么
2. 为什么这么选型
3. 遇到什么真实问题
4. 怎么定位和修复
5. 如果重来一次会怎么做

### 完成标志

你不是只完成了一个项目，而是积累了一套下次还能复用的方法论。

## 6. 通用的“先后顺序”细化规则

## 6.1 先写规范，再写代码

优先级顺序：

1. PRD
2. AGENTS
3. TECH_DESIGN
4. 测试
5. 骨架
6. 实现

如果反过来，AI 会在你还没定义边界时提前做大量错误决策。

## 6.2 先实现纯函数，再实现流程编排

对于复杂项目，优先顺序应该是：

1. 数据模型
2. 纯函数模块
3. 外部依赖适配器
4. 状态机/服务编排
5. API 层
6. UI 层

原因很简单：纯函数最容易测，最不容易飘。

## 6.3 先打通后端主链路，再优化前端体验

用户界面通常最显眼，但不是 vibecoding 的第一生产力。若主链路没打通，先做漂亮 UI 只会掩盖真实问题。

## 6.4 先 prompt 常量，再模板化

MVP 阶段：

1. Prompt 直接写 Python 常量
2. 流程跑通后，再迁移到 Jinja2 / YAML / Prompt 版本系统

原因：过早模板化会增加 AI 理解成本，降低改动速度。

## 6.5 先本地模式，再云模式

凡是支持本地模式的依赖，优先本地模式：

1. Qdrant 本地路径
2. DuckDB 内存库
3. 本地 JSON fixture
4. Docker Compose 服务

等逻辑稳定，再切云端或生产组件。

## 8. 给 AI 的协作模板

## 8.1 新功能开发模板

```text
请先阅读 AGENTS.md、PRD.md、TECH_DESIGN.md。
这次只做 [具体模块/功能]，不要扩散到其他模块。
先写测试，覆盖 happy path、边界 case 和一个异常 case。
再写最小实现使测试通过。
最后运行相关测试或最小验证命令，并汇报结果。
不要顺手重构无关代码，不要改 public API，除非文档要求。
```

## 8.2 调试问题模板

```text
请先定位 [具体现象/报错] 的控制路径。
先给出一个可证伪的本地假设，再做最小修改验证。
不要先全局搜索和大面积重构。
修复后运行最小相关测试或命令验证。
```

## 8.3 文档更新模板

```text
请根据当前实现，更新 [某个文档]。
要求文档与真实代码一致，避免写“未来计划”伪装成“已实现能力”。
如果发现文档和实现冲突，优先指出冲突点再修改。
```

## 8.4 骨架搭建模板

```text
请根据 PRD、AGENTS 和 TECH_DESIGN，先搭项目骨架。
只创建目录、入口、配置、模型、抽象接口和最小路由，不实现复杂业务逻辑。
每创建一个模块，都要保证它符合既定层级依赖规则。
```

## 9. 质量闸门

每个 vibecoding 项目都建议设 5 道质量闸门。

| 闸门 | 检查内容 | 不能通过时禁止做什么 |
| --- | --- | --- |
| G1 | PRD 清晰 | 禁止起技术栈和代码骨架 |
| G2 | AGENTS/TECH_DESIGN 成型 | 禁止批量生成代码 |
| G3 | 最小垂直切片跑通 | 禁止大规模扩功能 |
| G4 | 测试与安全基线达标 | 禁止接真实生产依赖 |
| G5 | 部署与运行文档齐全 | 禁止对外试用或上线 |

这 5 个闸门本质上是在防止 vibecoding 常见事故：**看起来进展很快，实际没有一个面向真实交付的闭环。**

## 10. Vibecoding 最常见的 12 个坑

1. PRD 没写清楚，AI 在错误目标上高速前进
2. 没有 AGENTS.md，AI 每次都按不同风格写
3. TECH_DESIGN 太空泛，代码落地时反复返工
4. 过早接真实外部依赖，分不清是代码问题还是环境问题
5. 没有 fixtures，测试数据每次都不一样
6. 让 AI 一次生成半个项目，结果没人能维护
7. 先做界面，主链路却没打通
8. Prompt 管理过度工程化，MVP 迭代速度被拖慢
9. 没有 release checklist，发版全靠感觉
10. 文档长期不更新，README 和真实实现完全不一致
11. 安全规则没有代码化，只停留在文档里
12. 没有 development journey，下一次还会踩同一批坑

## 12. 最终结论

一套通用的 vibecoding 构建顺序，本质上不是“先建后端还是先建前端”的问题，而是：

1. 先把目标收敛清楚
2. 先把 AI 约束清楚
3. 先把技术设计清楚
4. 先把测试地基打好
5. 先做一条最小垂直切片
6. 再逐步扩功能、接真依赖、补安全、做交付

只要你遵守这套顺序，vibecoding 就会从“拼运气”变成“可复制的工程流程”。
