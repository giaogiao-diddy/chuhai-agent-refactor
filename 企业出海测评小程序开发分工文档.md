# 企业出海测评小程序开发分工文档

## 1. 分工原则

当前项目建议由两位开发人员按照“前端小程序体验”和“微信云开发后端/业务逻辑”进行分工。

该项目表面上是测评问卷小程序，但核心复杂度集中在评分系统、用户分层、AI 报告生成、权限控制、报告解锁和顾问跟进。因此不建议简单按照页面数量平均拆分，而应按照职责边界拆分。

推荐分工：

1. 开发 A：负责小程序前端、用户端体验、顾问端界面和 UI 实现。
2. 开发 B：负责微信云开发、数据库、云函数、评分系统、AI 报告生成和权限控制。

## 2. 开发 A：小程序前端与用户体验

开发 A 主要负责用户端和顾问端所有可见页面，以及整体交互体验。

### 2.1 用户端功能

开发 A 负责以下用户端页面和交互：

1. 小程序基础框架搭建。
2. 微信登录授权页面。
3. 测评说明页。
4. 企业基础信息填写页。
5. 答题页。
6. 测评提交页/加载页。
7. 评分结果页。
8. 部分分析报告页。
9. 完整报告预览页。
10. 完整报告锁定页。
11. 客服微信解锁页。
12. 完整报告详情页。

### 2.2 答题交互

答题系统需要支持以下题型：

1. 单选题。
2. 多选题。
3. 填空题。
4. 说明型题目。

答题页需要实现：

1. 当前题目展示。
2. 答题进度条。
3. 上一题/下一题。
4. 必填校验。
5. 多选状态处理。
6. 填空内容保存。
7. 说明型题目展示。
8. 根据分层结果展示不同题目。
9. 提交前校验。

### 2.3 顾问端功能

开发 A 负责以下顾问端页面：

1. 顾问首页。
2. 用户测评列表页。
3. 用户详情页。
4. 答题详情页。
5. 评分详情页。
6. 报告详情页。
7. 顾问跟进报告页。
8. 跟进状态管理页。
9. 顾问备注页。

### 2.4 UI 重点

开发 A 需要重点保证：

1. 页面风格高端、专业、有视觉冲击力。
2. 测评说明页明确提示测评耗时 20-30 分钟。
3. 答题过程流畅，用户愿意完成较长测评。
4. 结果页有足够吸引力，让用户愿意添加客服微信解锁完整报告。
5. 报告页具有正式咨询报告的阅读体验。
6. 顾问端信息清晰，方便快速判断客户价值。

开发 A 的核心目标：

```text
让用户看得见、点得到、愿意继续，并愿意解锁完整报告。
```

## 3. 开发 B：微信云开发与业务逻辑

开发 B 主要负责云开发后端、数据库、云函数、评分计算、AI 报告生成、权限控制和顾问端数据接口。

### 3.1 云开发环境

开发 B 负责：

1. 微信云开发环境搭建。
2. 云数据库集合创建。
3. 云函数目录结构设计。
4. 云函数部署。
5. 云端权限规则配置。
6. 系统配置数据初始化。

### 3.2 数据库集合

开发 B 负责设计和维护以下集合：

1. users。
2. assessments。
3. questions。
4. reports。
5. consultant_notes。
6. system_config。

### 3.3 云函数

开发 B 负责以下云函数：

1. login。
2. getAssessmentConfig。
3. createAssessment。
4. submitAssessment。
5. calculateScore。
6. generateReports。
7. getUserReport。
8. unlockReportAfterWechatAdded。
9. getConsultantDashboard。
10. getConsultantAssessmentDetail。
11. updateFollowUp。
12. getSystemConfig。
13. updateSystemConfig。

### 3.4 评分系统

开发 B 负责实现两套评分系统：

1. 企业出海评分系统。
2. 顾问跟进评分系统。

评分系统需要支持：

1. 单选题按选项计分。
2. 多选题按选项累加计分。
3. 多选题可设置最高分上限。
4. 填空题可根据配置决定是否计分。
5. 说明型题目不计分。
6. 最终分数换算为 0-100 分。
7. 根据分数生成企业出海标签。
8. 根据分数生成顾问跟进标签。

### 3.5 AI 报告生成

开发 B 负责接入 AI 报告生成能力，并生成三类报告：

1. 部分分析报告。
2. 完整分析报告。
3. 顾问跟进报告。

AI 报告生成需要基于：

1. 企业基础信息。
2. 用户答题结果。
3. 用户分层结果。
4. 企业出海评分。
5. 顾问跟进评分。
6. 企业出海标签。
7. 顾问跟进标签。

### 3.6 权限控制

开发 B 负责权限控制：

1. 普通用户只能访问自己的测评和报告。
2. 普通用户不能访问顾问跟进评分。
3. 普通用户不能访问顾问跟进报告。
4. 顾问可以查看用户测评详情、评分和报告。
5. 管理员可以管理系统配置和顾问账号。
6. 所有敏感数据必须通过云函数校验权限。

开发 B 的核心目标：

```text
数据算得准、报告生得出、权限管得住。
```

## 4. 双方共同确认的接口

两位开发人员需要优先共同确认接口和数据结构，避免后续联调返工。

### 4.1 核心接口

建议优先确认以下接口：

```js
getAssessmentConfig()

submitAssessment({
  basicInfo,
  answers
})

getUserReport()

unlockReportAfterWechatAdded()

getConsultantDashboard()

getConsultantAssessmentDetail({
  assessmentId
})

updateFollowUp({
  assessmentId,
  status,
  note
})
```

### 4.2 需要共同确认的数据结构

双方需要共同确认：

1. 题目数据结构。
2. 答案提交格式。
3. 评分返回格式。
4. 标签返回格式。
5. 报告数据结构。
6. 用户解锁状态字段。
7. 顾问端列表筛选字段。
8. 错误码和加载状态。

## 5. 推荐开发顺序

建议按照以下顺序推进：

1. 两人共同确认数据结构和接口文档。
2. 开发 B 搭建云开发环境和数据库集合。
3. 开发 B 提供 mock 云函数或临时返回数据。
4. 开发 A 使用 mock 数据完成用户端完整流程。
5. 开发 A 使用 mock 数据完成顾问端页面。
6. 开发 B 完成真实评分逻辑。
7. 开发 B 接入 AI 报告生成。
8. 开发 A 接入真实接口。
9. 两人一起联调用户端完整链路。
10. 两人一起联调顾问端完整链路。
11. 两人一起测试权限、异常状态和报告解锁流程。
12. 最后统一做 UI 细节、数据校验和发布前测试。

## 6. 主链路优先级

项目开发过程中，应优先保证以下主链路跑通：

```text
登录
-> 填写企业信息
-> 答题
-> 提交测评
-> 计算评分
-> 生成标签
-> 生成报告
-> 用户查看部分报告
-> 用户添加客服微信
-> 系统自动解锁完整报告
-> 用户查看完整报告
-> 顾问查看用户详情和跟进报告
-> 顾问更新跟进状态和备注
```

## 7. 协作注意事项

### 7.1 开发 A 注意事项

1. 不需要等待所有后端接口完成后再开发页面。
2. 可以先使用 mock 数据完成页面和交互。
3. 需要提前和开发 B 确认接口字段。
4. 报告页要预留结构化内容展示能力。
5. 顾问端要优先保证信息密度和可读性。

### 7.2 开发 B 注意事项

1. 不需要参与过多 UI 细节。
2. 需要优先保证主链路接口可用。
3. 评分逻辑要可配置，方便后续替换正式题目和评分标准。
4. AI 报告生成结果要结构化存储。
5. 权限控制必须在云函数层完成。

## 8. 最终职责边界

最终职责可以概括为：

```text
开发 A 负责：用户体验、页面实现、交互流程、报告展示、顾问端界面。

开发 B 负责：云开发、数据库、云函数、评分系统、AI 报告、权限控制。
```
